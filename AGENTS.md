# Diff Generator - AI Agent Guide

## Project Structure

**Single-group layout (default):**
```
Chart.yaml                     Manages Chart Version as well as subchart dependency
values.yaml                    holds the default values if any overwride is not provided
tests/*                        used for testing tampltes and prom rules, sometimes contains values.yaml for testing only
templates/                     Additional helm templates, which is not covered by the subcharts,it cna be also be used for  addiotional features
README.*                       Its generated by code so dont try to update documentation only will be added as standard helm in values.yaml.
```

## Main functionality

This code will check if your values.yaml is compatible with the latest version of subcharts in your helm chart. if not then it will create a task in gitlab or somewhere.
 also the chart.yaml can be addiotionally provided to check diff between Chart.yaml and another file for example Chart-override.yaml. which can be new Chart.yaml with values for subchart and version modified.

### Additional Constraints
- A helm chart repo contains only **one `Chart.yaml`** — no need to scan for multiple charts.
- **`condition` and `tags`** on dependencies must be considered when resolving which subcharts are active.
- **Nested/transitive subchart dependencies** should be checked, but **only** when the transitive dependency has a direct relation (override) in the parent chart's `values.yaml`.

### Implementation Decisions
- **Language**: Go, using Helm SDK (`helm.sh/helm/v3`) for chart operations (no CLI exec).
- **Git operations**: Use `go-git` library — no shell commands for safety.
- **Delivery**: CLI tool invoked from GitLab CI.
- **On breaking change**: Create a **GitLab issue** with the detailed diff as a **markdown-formatted comment**.
- **OCI registries**: Not in initial scope, but architecture must support adding OCI later.
- **Auth for protected repos**: Environment variables (`HELM_REPO_USERNAME`, `HELM_REPO_PASSWORD`, `HELM_REPO_TOKEN`).

### Breaking Change Detection Rules
- **Structural change** (scalar → map, map → scalar, key depth change) → **BREAKING**
- **Keys removed upstream** that the parent `values.yaml` overrides → **BREAKING**
- **New keys added upstream** → **NOT breaking**. If the parent overrides the parent key and all its children, the new key should be noted (informational) but does not block the merge.
- **Value-only changes** (same key, same type, different value) → **NOT breaking**
- **Type changes** (e.g., `string "3"` → `integer 3`) → **NOT breaking** when the conversion is safe/lossless in most cases.
